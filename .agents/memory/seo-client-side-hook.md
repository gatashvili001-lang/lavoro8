name: SEO fixes for lavoro8 SPA (og-image, code splitting, per-route meta)
description: How the 3-part SEO task was resolved for the client-side-only lavoro8 Vite SPA
---

Lavoro8.it (`artifacts/lavoro8`) is a pure client-rendered Vite SPA with no server-side rendering. For the SEO cleanup:

- `src/lib/use-seo.ts` is the canonical per-page metadata hook (title, description, canonical, robots, OG/Twitter tags). Every route component should call it once with a `path` matching its route pattern; use `noindex: true` for account/admin pages (admin, profile) and for detail pages when the underlying entity isn't found (404-like states).
- Build-time SSG prerendering IS implemented for effectively all public routes, including DB-backed job/external-job detail pages: `scripts/prerender.ts` runs after `vite build`, fetches live job data from the api-server (`http://localhost:8080` by default, overridable via `PRERENDER_API_URL`) at build time, and writes a per-route `index.html` (real title/description/canonical/robots/OG/Twitter baked into the raw HTML) into `dist/public/<route>/index.html`. If the API is unreachable during a given build, job-detail prerendering is skipped gracefully (logs a warning, doesn't fail the build) and those pages fall back to CSR via `useSeo`. Known tradeoff: job listings are prerendered as of build time, so a listing removed/added between deploys won't be reflected in the static HTML until the next build — acceptable for a job board that rebuilds on every deploy.
- The prerender approach only works because this artifact's production deploy config (`.replit-artifact/artifact.toml`, `serve = "static"` with a `/* -> /index.html` rewrite) serves matching static files/directories before applying the SPA rewrite — i.e. a literal `dist/public/jobs/germany/index.html` wins over the catch-all rewrite for that path. Always check an artifact's `artifact.toml` production `serve`/`rewrites` config before assuming SSG output will actually be served.
- `og-image.png` never existed in `public/`; only `opengraph.jpg` does. Any new OG/Twitter image reference must point to `/opengraph.jpg`, not `/og-image.png`.
- Routes in `src/App.tsx` are lazy-loaded via `React.lazy` + `Suspense` (except `Home`, kept eager as the entry route) to enable route-level code splitting; `vite.config.ts` also defines `manualChunks` (clerk/react+radix/query/icons/vendor buckets) to shrink the single-bundle problem. A rollup "circular chunk" warning between `vendor`/`vendor-react` is benign (build succeeds) — didn't fully eliminate it since further splitting risked diminishing returns.
